## 2. 背景与目标

### 2.1 问题陈述

| 维度 | 内容 |
|------|------|
| **问题描述** | code-flow 对前端开发支持薄弱：① 设计文档模板（lite/full）后端/API 中心，无组件/状态/样式/交互维度；② 前端约束规范全是建议性，无机检；③ 注入 patterns 漏 `.vue`/`.svelte`/`app/` 等，编辑这些文件时根本不注入。同时，任务文档（prd/design/plan）扁平散落在 `tasks/<日期>/` 下，在"前后端设计分离"后难以集中管理与整体归档。 |
| **当前替代方案** | 前端开发者只能手动指定框架/手写规范；设计阶段套后端模板很别扭；一个需求的多份文档靠文件名后缀松散关联。 |
| **触发原因** | 前期分析确认前端适配缺口；用户提出前后端设计文档应分离，并按需求目录组织与归档。 |

### 2.2 目标与价值

| 维度 | 内容 |
|------|------|
| **核心目标** | 让任意前端/全栈项目跑完 init+learn 后，自动获得贴合本项目框架的前端规范注入、机检与设计文档；并将任务文档按需求目录组织，支持前后端 design 分离、plan 多 design 消费、整目录归档。 |
| **预期价值** | 前端"设计→编码"全链路被有效支撑且可强制；需求产物集中可管理。 |

**核心设计原则**：前端适配由 AI 命令**动态识别项目组成**完成（cf-init 检测、cf-learn 特化、align 扫描），**不硬编码框架清单**；人只维护框架中立脚手架。

**成功指标**

| 指标 | 当前值 | 目标值 | 衡量方式 |
|------|--------|--------|----------|
| 前端文件注入覆盖 | 仅 `.tsx/.jsx` | 按检测框架覆盖（含 `.vue`/`.svelte`/`app/`） | 端到端手验 |
| 前端最佳实践强制 | 无机检 | 分层/复用/样式-接口分离有机检并 cf-stop 守门 | 造例触发验证 |

## 4. 功能需求

### 4.1 功能清单

| 功能ID | 功能名称 | 功能描述 | 优先级 | 来源用户故事 |
|--------|---------|---------|--------|-------------|
| FEAT-01 | 前端注入覆盖动态化 | cf-init 按检测框架动态补 `frontend.patterns`（Vue→`**/*.vue` 等）；cf-learn 增加"注入覆盖漂移"检测并提示补齐 | P0 | US-01 |
| FEAT-02 | 独立前端设计文档 | 新增框架中立 `design-frontend.md` 脚手架（组件树/状态/数据流/UI 状态/样式/可访问性/交互验收）；align 动态产出前端设计文档 | P0 | US-02 |
| FEAT-03 | 前端最佳代码强制 | cf-learn 从证据动态生成前端分层/复用/样式-接口分离规范与机检 `checks`，接 quality_loop + cf-stop（按路径作用域控误报） | P0 | US-03 |
| FEAT-04 | 任务文档需求目录化 | prd/align/plan 写入 `tasks/<日期>/<需求>/`，文档同放一个需求目录 | P0 | US-04 |
| FEAT-05 | 前后端 design 分离 + plan 多 design 消费 | align 产出后缀区分的 `<req>.frontend.design.md`/`<req>.backend.design.md`；`/cf-task:plan <需求目录>` 自动发现全部 design，生成一份统一任务（Source 各自指向） | P0 | US-05 |
| FEAT-06 | 按需求目录归档 | cf-task:archive 将整个需求目录移入 `archived/<日期>/<需求>/`，兼容旧扁平布局 | P0 | US-06 |

#### FEAT-01: 前端注入覆盖动态化

- **描述**: 注入 patterns 由 AI 命令按检测结果动态写，而非静态硬编码框架清单。
- **验收标准**:
  - [ ] Vue demo `/cf-init` 后 `frontend.patterns` 自动含 `**/*.vue`；React demo 不含 `.vue`
  - [ ] cf-learn 检测到实际前端文件类型未被 patterns 覆盖时，作为候选提示补齐（走确认门）

#### FEAT-02: 独立前端设计文档

- **描述**: 新增框架中立前端设计模板，align 依据自身代码库扫描动态选用并产出。
- **验收标准**:
  - [ ] `design-frontend.md` 存在且被 cf-init 列入 shared 模板、注册到 `config.yml`（`tags:[]` 非注入）
  - [ ] align 对前端/全栈需求自动选用该模板产出前端设计文档（无需人为 flag）

#### FEAT-03: 前端最佳代码强制

- **描述**: cf-learn 从项目真实框架/lint 配置/代码证据生成前端规范与机检，接最强强制链路。
- **验收标准**:
  - [ ] cf-learn 能从真实组件/services 生成分层/复用/样式-接口分离候选 + checks 草稿
  - [ ] 组件内裸 `fetch`/`axios` 触发机检；`services/` 层内同样写法不触发（路径作用域精度）
  - [ ] 违规项在会话收尾被 cf-stop 拦截

#### FEAT-04: 任务文档需求目录化

- **描述**: prd/align/plan 写入 `tasks/<日期>/<需求>/`，集中一个需求的全部文档。
- **验收标准**:
  - [ ] `/cf-task:prd` 生成 `tasks/<日期>/<需求>/<需求>.prd.md`
  - [ ] align/plan 写入同一需求目录

#### FEAT-05: 前后端 design 分离 + plan 多 design 消费

- **描述**: align 后缀区分前后端 design；plan 指定需求目录自动发现多 design 生成统一任务。
- **验收标准**:
  - [ ] 全栈需求 align 产出 `*.frontend.design.md` + `*.backend.design.md`
  - [ ] `/cf-task:plan <需求目录>` 自动发现两份 design，生成一份 `<需求>.md`，FE/BE 任务 Source 各自指向

#### FEAT-06: 按需求目录归档

- **描述**: archive 将整个需求目录归档，兼容旧扁平布局。
- **验收标准**:
  - [ ] `/cf-task:archive <需求>` 将整个需求目录移入 `archived/<日期>/<需求>/`
  - [ ] 旧扁平布局任务仍可被查找与归档（逐文件回退）

### 4.2 边缘情况

| 场景 | 预期行为 |
|------|----------|
| 纯后端/纯前端需求 | align 只产出对应单域 design；无关域不生成 |
| 非前后端需求（CLI/脚本） | 使用 `<req>.design.md`（无域后缀） |
| 旧扁平布局任务 | start/status/plan/archive 仍可经 `tasks/**` glob 命中并归档 |

## 5. 非功能需求

| 指标ID | 类型 | 要求 |
|--------|------|------|
| NFR-COMPAT-01 | 兼容性 | 新旧任务布局并存；旧扁平文档（含 `archived/`）继续可查找/归档 |
| NFR-COMPAT-02 | 兼容性 | 跨平台命令对等（claude/costrict/opencode/codex），`test_adapter_parity` + codex ≥85% 通过 |
