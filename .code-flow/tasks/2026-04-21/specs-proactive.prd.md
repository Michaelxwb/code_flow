## 2. 背景与目标

### 2.1 问题陈述

| 维度 | 内容 |
|------|------|
| **问题描述** | 当前 specs 注入完全由「文件路径 + prompt 关键词」决定，导致三类可观察的问题：① **注入不对称**：新建文件与修改文件拿到同一组规范，quality/test 类规则常在新建场景缺席；② **会话无记忆**：上一会话做过的关键决策（如某个边界取舍）在新会话中丢失，AI 重复询问或给出不一致建议；③ **规范演进滞后**：项目中已形成的新实践（如错误处理约定）停留在代码里，未沉淀到 specs，新会话无法复用 |
| **当前替代方案** | `cf-learn --review` 可手动分析变更并建议沉淀，但需作者定期主动调用；实际使用中运行频率 < 1 次/周，远低于变更节奏 |
| **触发原因** | code-flow 已具备被动注入闭环，继续演进方向是让"判断 / 记忆 / 沉淀"也自动化，从「会读规范」走向「会用 + 会写规范」 |

---

### 2.2 目标与价值

| 维度 | 内容 |
|------|------|
| **核心目标** | 在现有被动注入之上，叠加三项能力：**意图感知的注入增强**、**跨会话的决策记忆**、**从变更中派生候选规范**；默认安全降级，单项失败不影响主流程 |
| **预期价值** | ① 新建/修改/删除场景拿到差异化规范；② 会话间关键决策可被下一次会话引用；③ 规范库跟随代码演进，减少手工维护 |

**术语口径**：本文档中"主动"特指「无需用户显式触发即可执行」。AI 产出候选、**用户确认**后落盘的流程，仍视为主动（对比：完全自动落盘 = 全自动，纯手工触发 = 被动）。

**成功指标**

| 指标 | 当前基线（v1.1 采集） | 目标值 | 衡量方式 |
|------|---------------------|--------|----------|
| **注入场景差异度** | 0%（新建 vs 修改注入集合相同） | ≥40% 的差异 | 对比相同文件在 `create` / `modify` 下注入 spec 集合的 Jaccard 距离 |
| **会话决策复用率** | 不可度量（无记录） | ≥30% 的新会话在 SessionStart 摘要中存在决策条目 | FEAT-03 产出的摘要 JSON 中 `key_decisions` 数组非空的会话占比（由 `cf-stats` 聚合） |
| **候选规范采纳率** | ~0%（基本无手工运行） | ≥50% 候选被用户确认落盘 | `cf-learn --review` 采纳/生成比 |
| **误增强率（质量护栏）** | N/A | ≤10% | 意图增强前后注入集合差集（新增 spec 条目）中，在后续该会话内**未被引用**（即 spec 内容未出现在 AI 输出中）的比例；由 `cf-stats --session` 离线计算 |

> 基线 0% 的项在 v1.1 实施前先埋点采集 1 周作为锚点；`cf-stats` 增加对应聚合。

## 4. 功能需求

### 4.1 功能清单

| 功能ID | 功能名称 | 功能描述 | 优先级 | 来源用户故事 |
|--------|---------|---------|--------|-------------|
| FEAT-01 | 意图感知增强 | Hook 触发时识别操作类型（create/modify/delete）与风险层级，据此调整注入的 tag 集合 | P0 | US-01 |
| FEAT-02 | 会话活动日志 | 在会话范围内记录注入事件与关键决策，供总结与复用 | P0 | US-02 / US-03 |
| FEAT-03 | 跨会话摘要 | 新会话开始时，呈现上一会话的摘要（关键决策 + 触及的 domain + 注入集合） | P0 | US-02 |
| FEAT-04 | 候选规范派生 | 基于工作区变更生成候选规范清单，过滤已有规则后由用户二次确认 | P0 | US-03 |
| FEAT-05 | 降级与可观测 | 任一新能力异常时自动跳过并记录，不阻塞被动注入；`cf-stats` 暴露相关计数 | P0 | US-04 |
| FEAT-06 | 能力开关 | 每项新能力（FEAT-01/02/03/04）在 config 中可独立启用/关闭；FEAT-02 默认开启，关闭后 FEAT-03 自动跟随关闭 | P1 | US-05 |

---

#### FEAT-01: 意图感知增强

- **描述**: 基于文件是否存在、Hook 工具类型（Write/Edit/Read）推断操作意图，进而调整 tag 匹配集合
- **验收标准**（用户可观察）:
  - [ ] 在**新建**一个 Python 脚本时，注入结果中至少包含一条来自 `test` 或 `quality` 域的规范（若 specs 中存在）
  - [ ] 在**修改**已有文件时，注入结果与 v1.0 被动匹配保持 ≥80% 重合（避免剧烈漂移）
  - [ ] 用户可通过 `cf-stats` 查看每次注入识别到的 `action_type`
  - [ ] 注入 JSON 对外 schema 保持不变（向后兼容）

---

#### FEAT-02: 会话活动日志

- **描述**: 以 `session_id` 为维度持久化注入与决策事件，作为 FEAT-03 / FEAT-04 的数据源
- **"关键决策"数据来源**：日志中的决策事件由以下两类来源自动写入：① **Hook 自动记录**——每次 PreToolUse/UserPromptSubmit 注入时，记录注入的 spec 集合和 action_type（无需用户干预）；② **SessionStart 推断**——会话结束时（下次 SessionStart 触发时），由摘要器从注入序列中推断出"高频触及的 domain + spec"作为候选决策点。不依赖用户手工标记。
- **验收标准**（用户可观察）:
  - [ ] 每次注入后，通过 `cf-stats --session <id>` 可列出该会话的注入事件（文件、domain、spec、tags、action_type）
  - [ ] 日志保留策略有明确上限（默认 30 天 / 超过 **5MB** 触发滚动清理）
  - [ ] 日志损坏或写失败时，主注入流程不受影响（见 FEAT-05）

---

#### FEAT-03: 跨会话摘要

- **描述**: 新会话启动时，从上一会话日志派生摘要，作为上下文呈现给 AI
- **验收标准**（用户可观察）:
  - [ ] 第二次会话启动时，在 SessionStart 注入内容中能看到上一会话的：duration、触及 domain 集合、关键决策点（若有）
  - [ ] 首次运行或上一会话无记录时，呈现"无前置摘要"占位而非报错
  - [ ] 摘要体积 ≤ 2KB（避免挤占注入预算）

---

#### FEAT-04: 候选规范派生

- **描述**: 扫描 `git diff`（工作区 + 最近提交窗口可配置），提取可能的新约定，经去重过滤已有 specs 后产出候选清单
- **验收标准**（用户可观察）:
  - [ ] `cf-learn --review` 运行后输出候选清单，每条包含：domain、描述、来源文件、置信度
  - [ ] 候选清单中**不出现**已存在于 specs 的规则（文本相似度判重）
  - [ ] 空 diff 或全部已覆盖时，明确输出"无新候选"而非空文件
  - [ ] 支持 `--auto` 标志用于 CI/定时场景（仍只是"生成候选"，不自动落盘）

---

#### FEAT-05: 降级与可观测

- **描述**: 新能力全部为可失败的**增强层**；失败时记录错误但不向 Hook 抛出
- **验收标准**（用户可观察）:
  - [ ] 人为构造 FEAT-01/02/03/04 任一模块的异常（如日志目录不可写），主注入流程仍返回 v1.0 等价结果
  - [ ] `cf-stats` 暴露每项新能力的 `enabled` / `degraded_count` / `last_error`
  - [ ] 降级不输出到 stdout（保持 Hook 协议 JSON-only）

---

#### FEAT-06: 能力开关

- **描述**: `config.yml` 中为每项新能力提供独立开关；遵循现有 `resolve_compress` 风格（仅 literal `false` 禁用）
- **验收标准**（用户可观察）:
  - [ ] 关闭**意图感知增强**能力后，注入行为与 v1.0 完全一致（可用 `cf-scan` 对比验证）
  - [ ] 关闭**会话日志**后，跨会话摘要能力随之不可用，且有明确提示
  - [ ] 未配置字段时使用安全默认（见术语表）
  - [ ] 配置变更无需重启，下一次 Hook 调用生效

---

### 4.2 边缘情况

| 场景 | 预期行为 |
|------|----------|
| 意图分析无法判定（路径既不存在也非工具提示） | 回退到 v1.0 被动匹配，不增强 tag |
| 上一会话日志损坏（JSON 解析失败） | SessionStart 跳过摘要，日志文件移至 `.session-log.json.corrupt.<ts>` 备份 |
| 候选规范数量爆炸（单次 >50 条） | 按置信度截断前 50 条，提示总数 |
| session-log 文件膨胀（>5MB） | 触发滚动，保留最近 30 天，更老的归档到 `sessions/YYYY-MM/` |
| 非 git 仓库或 git 不可用 | FEAT-04 跳过并在 `cf-stats` 记录原因，不影响其他 FEAT |

## 5. 非功能需求

| 指标ID | 类型 | 要求 |
|--------|------|------|
| NFR-PERF-01 | 性能 | PreToolUse Hook P95 响应时间 ≤ **当前基线 + 150ms**（当前基线在 v1.1 实施前采集，预计 <500ms） |
| NFR-PERF-02 | 性能 | SessionStart 摘要生成 P95 ≤ 300ms |
| NFR-PERF-03 | 资源 | `.session-log.json` 单文件稳态 ≤ **5MB**；超出触发滚动 |
| NFR-RELIABILITY-01 | 可靠性 | 新能力任一异常不得导致 Hook 退出码 ≠ 0 或输出非 JSON |
| NFR-ACCURACY-01 | 准确性 | 意图增强误增强率 ≤ 10%（见 2.2 指标口径） |
| NFR-COMPAT-01 | 兼容性 | 不引入 Python 标准库以外的依赖；Hook JSON schema 保持向后兼容 |
| NFR-SEC-01 | 数据安全 | 所有新增数据仅落本地 `.code-flow/`；不外传、不写入代码仓外路径 |

## 6. 范围与边界

| 类别 | 内容 |
|------|------|
| **范围（In Scope）** | ① 意图感知增强（基于路径/工具/存在性的规则推断，**不含 LLM 语义分析**）；② 会话日志与上一会话摘要（仅 N-1 → N，**不做**多会话聚合学习）；③ 候选规范派生（仅**产出候选**，**不自动落盘**）；④ 每项能力独立开关与降级；⑤ 配套单元测试与 `cf-stats` 指标 |
| **非范围（Out of Scope）** | ① LLM / 向量相似度用于意图分析；② 跨多会话的自适应权重学习；③ 候选规范自动落盘到 specs；④ UI 展示（仅 CLI 输出）；⑤ 远程/多人共享会话日志 |
| **前置假设** | ① Claude Code / Costrict 提供稳定的 `session_id` 或可由 PID 回退（已有 `resolve_session_id`）；② SessionStart Hook 事件可用；③ 项目使用 git；④ 规则驱动的意图识别在 code-flow 场景下可达到 NFR-ACCURACY-01（需 v1.1 实施前小样本验证） |

> **自洽性说明**：FEAT-03 的"上一会话摘要"是 N-1 → N 的**单跳引用**，不构成 Out-of-Scope 中的"跨会话自适应学习"；后者特指权重/参数随会话变化。

## 7. 依赖与风险

### 7.1 外部依赖

| 依赖方 | 依赖内容 | 风险等级 | 说明 |
|--------|----------|---------|------|
| Claude Code / Costrict Hook 协议 | SessionStart 事件、stdin JSON schema 稳定 | 中 | 上游协议变更需适配 |
| git (≥2.0) | FEAT-04 需 `git diff` 输出稳定 | 低 | 回退：git 不可用时跳过 FEAT-04 |
| Python ≥ 3.8 标准库 | 无第三方依赖约束 | 低 | 已是项目既有约束 |

> 内部模块改造（`cf_core.py` / `cf_inject_hook.py` / `cf_session_hook.py`）不是"依赖方"，将在 design 阶段的架构设计中说明。

### 7.2 风险识别

| 风险ID | 描述 | 影响 | 缓解方案 |
|--------|------|------|---------|
| R-01 | 规则驱动的意图分析准确率不达 NFR-ACCURACY-01 | 中 | 先做 1 周埋点采样验证；不达标时 FEAT-01 默认关闭，仅留配置 |
| R-02 | session-log 写频繁影响 Hook 延迟 | 中 | 通过非阻塞写入保证主路径延迟不受影响；超时后降级跳过写入（具体超时值在 design 阶段确定） |
| R-03 | 候选规范产生噪音导致用户疲劳 | 中 | 置信度阈值过滤 + 默认不自动运行 + 单次上限 50 条 |
| R-04 | 新能力与现有压缩/预算逻辑冲突 | 低 | FEAT-03 摘要纳入 `select_specs_tiered` 预算，统一计费 |
| R-05 | 上游 Hook 协议变更破坏 session_id | 低 | 继续走 `resolve_session_id` PID 回退 |

## 附录：术语表

| 术语 | 定义 |
|------|------|
| PRD | Product Requirements Document，产品需求文档 |
| US | User Story，用户故事 |
| FEAT | Feature，功能项 |
| NFR | Non-Functional Requirement，非功能性需求 |
| action_type | 操作类型：`create` / `modify` / `delete` / `read` |
| 主动 | 无需用户显式触发即执行的能力（即便最终需用户确认结果） |

| 安全默认 | 新能力字段缺失时的默认策略：FEAT-01 **开启**；FEAT-04 `auto_analyze` **关闭** |
| N-1 → N 单跳 | 新会话仅引用紧邻的上一会话摘要，不跨多层 |
