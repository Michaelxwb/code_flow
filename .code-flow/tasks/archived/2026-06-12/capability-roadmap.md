# code-flow 能力差距评估与提升路线图

> **文档编号**: ROADMAP-2026-06-12-001
> **创建日期**: 2026-06-12
> **来源**: 开发者视角能力评估对话（四维度：优雅高效 / 符合规范 / 高质量 / 智能 spec）
> **关系**: 与 PRD-2026-04-21-001（FEAT-01~06）互补——PRD 管"记忆与派生"，本路线图补"两个闭环"

## 核心判断

**code-flow 目前是"事前提醒系统"：注入即结束。** 最大缺口是两个闭环：

1. **合规闭环**——spec 注入后没有任何机制检查 agent 是否遵守。spec 是"建议"，违反零后果。约束 agent 写出优雅/规范/高质量代码，三个诉求共享这一个答案。
2. **学习信号闭环**——用户对 agent 的纠正（prompt 中的否定句式、人工改掉 agent 写的代码）是最高价值的规范证据，目前全部丢弃。cf-learn 依赖人主动运行（实际 <1 次/周）。

## 差距清单

### 合规闭环

| 编号 | 缺口 | 说明 |
|------|------|------|
| G1 | **PostToolUse 合规反馈**（核心） | spec 条目标注可机检性（`check: regex/ast/cmd`），Edit/Write 后对 diff 做轻量检查，违规经 `additionalContext` 即时反馈 agent 当场修正。把 spec 从"建议"变"闭环" |
| G2 | **Stop 收尾守门** | validation.yml 体系已存在但纯手动（/cf-validate）；接入 Stop hook，会话收尾自动跑变更文件关联的 lint/test |
| G3 | **spec 内容形态升级** | 规则清单 → ✅/❌ 好坏代码对照示例（few-shot steering）。"优雅"靠示例引导比靠条文约束有效 |
| G4 | **违规度量** | 记录哪条规则被违反多少次 → cf-stats 展示 Top 违规规则。常违反 = 规则写得不清楚或不合理，反向指导规则改写 |

### 学习信号闭环

| 编号 | 缺口 | 说明 |
|------|------|------|
| G5 | **纠正信号采集** | UserPromptSubmit 识别否定/纠正句式（"不要/不对/我说过/改回去"）记入会话日志；git 检测"agent 写入后被人工改掉"的文件。两类信号喂给 cf-learn 候选流——从"定期扫代码"升级为"从被纠正中学习" |
| G6 | **spec 有效性度量与淘汰** | 长期不命中、注入后违规率不降、反复被用户否定的 spec 标记"待复审"。spec 库要会做减法，否则预算被僵尸规则占满 |
| G7 | **注入准确性** | （已独立立项实施：方案 C 目录+自取重构，见 2026-06-12 注入准确性重构） |

### 体系联动

| 编号 | 缺口 | 说明 |
|------|------|------|
| G8 | **任务-规范联动** | cf-task 与 spec 注入互不通信。`cf-task:start` 时把 design 文档的验收标准/约束节转为本会话临时 spec 注入——"这个任务的 spec"比"这个目录的 spec"更贴上下文 |

## 风险提示

- PostToolUse 增加编辑延迟：checks 限 regex 级 + 超时跳过（沿用 NFR-PERF-01 思路）
- 误报骚扰：只提示不阻断；G4 度量反向监控误报率，常误报 check 自动降级
