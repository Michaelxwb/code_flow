## 2. 背景与目标

### 2.1 问题陈述 [必填]

| 维度 | 内容 |
|------|------|
| **问题描述** | code-flow 目前是"事前提醒系统"：spec 注入即结束，存在两个闭环缺失。① **合规闭环缺失**——注入后没有任何机制检查 agent 是否遵守，spec 是"建议"，违反零后果，用户只能在 code review 时人工兜底；② **学习信号闭环缺失**——用户对 agent 的纠正（prompt 否定句式、人工改掉 agent 写的代码）是最高价值的规范证据，目前全部丢弃，规范库演进完全依赖人工 |
| **当前替代方案** | 合规侧：`/cf-validate` 手动触发校验，依赖用户记得运行；学习侧：`cf-learn --review` 手动扫描变更，实际运行频率 <1 次/周，远低于纠正发生频率 |
| **触发原因** | G7 注入准确性重构已交付（catalog 模式，可达率 76%→100%），"注入准"的地基已就位；继续演进方向是"注入后管不管用"（合规）与"规范库会不会成长"（学习）。同时项目面向公开 npm 用户发布，新能力必须默认安全、可降级 |

---

### 2.2 目标与价值 [必填]

| 维度 | 内容 |
|------|------|
| **核心目标** | 在 catalog 注入地基之上建立两个闭环：**合规闭环**（违规当场反馈修正）与**学习闭环**（纠正信号自动沉淀为候选规范），并以公开 npm 用户可用的标准交付（零配置、自动降级、误报可关） |
| **预期价值** | ① agent 违规当场修正，减少 code review 人工兜底；② 用户纠正一次即沉淀，不再重复说第二遍；③ 规范库随使用自动演进且会做减法；④ 公开用户开箱即用，出错不打扰 |

**成功指标**（埋点先行：基线指标在 FEAT-01/02 上线后采集 2 周锚定，目标值在基线报告中补充评审；护栏值随本 PRD 生效）

| 指标 | 当前基线 | 目标值 | 衡量方式 |
|------|---------|--------|----------|
| 违规当场修正率 | 不可度量（无反馈机制） | 基线采集后定 | 会话日志中"违规反馈事件后 N 个工具调用内同文件再次编辑且检查通过"的占比，`cf-stats` 聚合 |
| 误报率（护栏） | N/A | **≤10%**（硬上限） | 被用户显式忽略/标记误报的反馈事件占比；超限的 check 自动降级（见 FEAT-08） |
| 纠正信号捕获-采纳率 | ~0%（信号全部丢弃） | 基线采集后定 | cf-learn 候选中来源为纠正信号的条目被用户确认落盘的比例 |
| 僵尸规范识别数 | 不可度量 | 基线采集后定 | 被标记"待复审"且用户确认淘汰/改写的 spec 条目数，`cf-stats` 聚合 |
| Hook 延迟增量（护栏） | 当前 PostToolUse 无注册（0ms） | **P95 增量 ≤150ms**（硬上限） | CF_DEBUG 埋点对比启用前后 hook 耗时分布 |

## 4. 功能需求

### 4.1 功能清单 [必填]

| 功能ID | 功能名称 | 功能描述 | 优先级 | 来源用户故事 | 路线图 |
|--------|---------|---------|--------|-------------|--------|
| FEAT-01 | PostToolUse 合规反馈 | agent 编辑文件后对变更做可机检规则检查，违规即时反馈 agent 修正 | P0 | US-01, US-02 | G1 |
| FEAT-02 | 发布护栏（开关/降级/误报治理） | 每项新能力独立开关、异常自动降级、误报标记与自动降敏 | P0 | US-02, US-03 | 横切 |
| FEAT-03 | Stop 收尾守门 | 会话收尾自动执行 validation.yml 中变更文件关联的校验 | P1 | US-04 | G2 |
| FEAT-04 | 纠正信号采集 | 识别 prompt 纠正句式与"agent 写入后被人工改掉"信号，喂给 cf-learn 候选流 | P1 | US-05 | G5 |
| FEAT-05 | 违规度量 | 按规则聚合违规次数与趋势，cf-stats 输出 Top 违规榜 | P1 | US-06 | G4 |
| FEAT-06 | spec 有效性度量与复审标记 | 长期不命中/违规率不降/反复被否定的 spec 条目标记"待复审" | P1 | US-07 | G6 |
| FEAT-07 | spec 示例化模板 | spec 模板与 cf-learn 输出支持 ✅/❌ 好坏代码对照示例 | P2 | US-08 | G3 |
| FEAT-08 | 任务-规范联动 | cf-task:start 将 design 文档验收标准转为会话级临时约束注入 | P2 | US-09 | G8 |

> P0=核心必做，P1=重要，P2=增强。分期交付：P0 → P1 → P2 三个里程碑，每期独立可发布。

#### FEAT-01: PostToolUse 合规反馈

- **描述**: spec 约束条目可标注机检方式（regex/ast/命令三级），Edit/Write 后对变更内容执行匹配 spec 的检查，违规经 `additionalContext` 反馈 agent 当场修正；只提示不阻断
- **验收标准**:
  - [ ] 在含 `checks` 标注的项目中，agent 写入违规代码（如 `print()` 调试）后，下一轮上下文中出现违规说明与对应规则原文
  - [ ] 反馈后 agent 修正的行为可在会话日志中追溯（依赖 PRD-2026-04-21 FEAT-02）
  - [ ] 无 `checks` 标注的 spec 行为与现状完全一致（纯注入，零开销）
  - [ ] 检查超时（上限见 NFR-PERF-01）自动跳过并记录，不阻塞工具调用

#### FEAT-02: 发布护栏（开关/降级/误报治理）

- **描述**: 面向公开用户的横切能力：config 中每项新能力独立开关（缺省安全值）；任一能力异常自动降级为 no-op 并计数；用户可对单条 check 标记误报，误报率超护栏的 check 自动停用
- **验收标准**:
  - [ ] 全新安装零配置即获得默认行为（FEAT-01 默认开，FEAT-04 采集默认开、落盘必须确认）
  - [ ] 人为构造任一能力异常（日志不可写/规则损坏），主注入与工具调用流程不受影响，`cf-stats` 暴露 `degraded_count` / `last_error`
  - [ ] 单条 check 可通过 config 一键停用；被标记误报达护栏阈值的 check 自动停用并在 cf-stats 标注
  - [ ] 升级不覆盖用户自定义配置与规范（沿用 fileCategory user 类保护）

#### FEAT-03: Stop 收尾守门

- **描述**: 会话收尾（Stop hook）时按 validation.yml 的 trigger globs 对本会话变更文件执行 lint/test 校验，未过项作为反馈呈现；平台不支持 Stop 事件时降级跳过
- **验收标准**:
  - [ ] 会话内编辑过的文件命中 trigger 时，收尾自动执行对应命令并反馈结果
  - [ ] 校验全过时静默（无 stdout 噪音）；超时跳过并记录
  - [ ] codex/opencode 等无 Stop 事件的平台自动降级，cf-stats 标注原因

#### FEAT-04: 纠正信号采集

- **描述**: 用户的修正全部通过与 agent 对话发生（不人工改代码），因此信号以对话为中心：① UserPromptSubmit 识别纠正句式（"不要/不对/我说过/改回去"等，中英文），产生 correction 事件；② 将纠正语句与 agent **随后的修正编辑**（会话日志中同会话紧邻的 Edit/Write 事件）配对，形成"纠正原因 + 修正前后对照"的证据对——天然适配 FEAT-07 的 ✅/❌ 示例格式。cf-learn 聚合为带证据的候选规范，仍需用户确认落盘
- **验收标准**:
  - [ ] 含纠正句式的 prompt 在会话日志中产生 correction 事件（含原句与上下文文件）
  - [ ] 纠正语句后 agent 在同会话内修正文件时，correction 事件关联该修正编辑，候选证据含修正前后对照
  - [ ] 纠正语句后无后续编辑（纯讨论）时，仅保留语句信号，不强行配对
  - [ ] `cf-learn --review` 输出的候选中，来源为纠正信号的条目标注信号原文与发生次数
  - [ ] 同类纠正聚合计数，不重复生成候选；所有数据仅存本地 `.code-flow/`（NFR-SEC-01）

#### FEAT-05: 违规度量

- **描述**: 基于会话日志按规则聚合违规事件，`cf-stats` 输出 Top 违规规则榜（次数/趋势/涉及文件域）
- **验收标准**:
  - [ ] `cf-stats` 可列出违规次数 Top N 规则及其所属 spec
  - [ ] 数据窗口可配置（默认 30 天，与日志滚动策略一致）
  - [ ] 无日志数据时输出"暂无数据"而非报错

#### FEAT-06: spec 有效性度量与复审标记

- **描述**: 聚合三类信号——长期未命中注入、注入后违规率未下降、被纠正信号反复否定——为 spec 条目打"待复审"标记，`cf-scan` 输出复审清单；标记仅供人工决策，不自动删除
- **验收标准**:
  - [ ] `cf-scan` 输出待复审条目及其触发原因（未命中天数/违规趋势/否定次数）
  - [ ] 阈值可配置且有缺省安全值
  - [ ] 用户处理（改写/删除/豁免）后标记清除，豁免项不再重复提示

#### FEAT-07: spec 示例化模板

- **描述**: spec 模板增加 ✅/❌ 代码对照示例段，cf-learn 生成候选时优先以示例对形式表达；注入压缩保留示例完整性
- **验收标准**:
  - [ ] 模板含示例段且 cf-init 部署的骨架 spec 演示该写法
  - [ ] cf-learn 候选输出包含从实际 diff 提炼的 ✅/❌ 对照（可由用户编辑）
  - [ ] `compress_content` 不破坏代码块（现有无损变换已保证，回归覆盖）

#### FEAT-08: 任务-规范联动

- **描述**: `cf-task:start` 激活子任务时，将其 design 文档的验收标准/约束节提取为会话级临时 spec，按 catalog 模式规则注入；任务归档时自动失效
- **验收标准**:
  - [ ] start 后，该任务的验收标准在 Spec Catalog 中以临时条目出现
  - [ ] 任务 design 无验收节时静默跳过
  - [ ] archive 后临时条目消失，不污染常驻规范库

### 4.2 边缘情况 [可选]

| 场景 | 预期行为 |
|------|----------|
| spec 的 `checks` 标注语法错误 | 跳过该 check 并在 cf-scan 报告，不影响其他检查与注入 |
| 检查命令在用户环境不存在（如未装 linter） | 该 check 降级跳过并计数，不报错打扰 |
| 纠正句式误判（用户说"不要紧"） | 句式词表保守收录 + 聚合阈值（单次信号不生成候选）；误判仅产生本地日志事件，无用户可见噪音 |
| 纠正语句实为需求变更而非规范纠正（"改成蓝色"） | 信号仅作为候选证据之一，cf-learn 标注置信度，落盘必须用户确认 |
| 纠正后 agent 的修正编辑跨多个文件 | 配对取同会话窗口内的全部后续编辑，证据中按文件列出，由 cf-learn 截取相关段 |
| 会话日志膨胀 | 沿用 PRD-2026-04-21 策略：30 天 / 5MB 滚动 |
| 平台无 PostToolUse/Stop 事件 | 按平台能力矩阵降级：仅享受日志与学习，不享受该 hook 反馈；cf-stats 标注 |
| 用户项目无 validation.yml | FEAT-03 静默跳过 |

## 5. 非功能需求

| NFR-PERF-01 | 性能 | PostToolUse 检查 P95 增量 ≤150ms；单 check 超时上限（缺省 2s，可配）超时即跳过 |

| NFR-SEC-01 | 数据安全 | 纠正信号、违规事件等所有新增数据仅存本地 `.code-flow/`，不外传；文档中显式声明（公开用户信任前提） |

| NFR-COMPAT-02 | 兼容性 | 存量用户升级后默认行为变化仅限新增反馈提示；所有新能力可经 config 整体关闭回到 v0.5 行为 |
